### 1c. `kb_sync.py`

Incremental sync driven by `.geppetto_manifest.json` in `./docs/`.
Manifest entry per file: `{sha256, mtime, size, chunk_ids[], ingested_at, tier}`.

On `sync()`:
1. Enumerate `./docs/` for supported extensions.
2. Classify: NEW / MODIFIED (hash changed) / SKIP (mtime+size fast-path) / DELETED.
3. NEW/MODIFIED: delete old chunks, re-chunk (~800 chars + overlap), re-embed,
   update manifest. DELETED: remove chunks + manifest entry.
4. Run **metric extraction** (Â§1d) on authoritative files â€” results go to
   **pending queue**, not directly to fact store.
5. Return `{new, modified, deleted, skipped, pending_extractions}` counts.

Atomic manifest writes (temp file + rename) â€” guards against crash mid-sync.
Use Chroma's local embeddings â€” no per-file OpenAI cost, works offline.

Wire-in: runs **blocking at server startup** (one-line summary printed) before
the app accepts requests. Also exposed as **`POST /api/sync`** for manual
re-trigger from the dashboard.

### 1d. Auto-discovered metric catalog

The metric catalog is **not hardcoded**. When a document is synced, a Haiku pass
over the text extracts measurable facts:

```
doc text â†’ Haiku â†’ [{metric_key, value, unit, as_of, source}, ...]
```

Metric keys are generated by the LLM using a consistent `domain.attribute`
naming convention (e.g., `qa.tests_passed_pct`, `release.date`).

**Key stability (load-bearing).** Auto-discovery only works if the *same concept*
gets the *same key* across documents and runs. If keys drift
(`qa.tests_passed_pct` one day, `qa.test_pass_rate` the next) the timeline
fragments into half-series and OUTDATED detection silently breaks within each.
To hold keys stable:
- Inject the **current list of known keys** into every extraction prompt â€” "reuse
  an existing key if the concept matches; mint a new one only if none fits."
- Normalize/alias returned keys (lowercase, canonical synonyms) before storing.
- Let the PM **merge / rename keys** in the pending tab; persist aliases so future
  extractions of a renamed concept resolve to the canonical key.

This replaces the hardcoded `METRIC_CATALOG` in `facts.py` with an open registry
that grows with the project â€” and naturally handles new risks, new tasks, and
new project areas as they appear in documents.

### 1e. Human-in-the-loop: provisional facts + pending queue

Auto-extracted metrics are never silently granted full trust, but handling differs
by tier to avoid a **false-alarm window** (Guardrail G3). Holding *everything*
pending would mean a correct new claim gets flagged CONTRADICTED while the update
waits for review; auto-accepting *everything* would let a Haiku misread ("6 to 8"
â†’ "628") become truth. The split below avoids both:

- **Authoritative docs â†’ committed as `provisional`.** The value goes live
  immediately (no false-alarm window) but is marked `provisional` and used in
  validation at **reduced confidence with an "unconfirmed" note** in the alert. The
  PM **Confirms** (â†’ full trust), **Edits**, or **Rejects** it in the Pending tab.
- **Derived (AI meeting notes) â†’ held in pending.** Nothing enters the fact store
  until the PM explicitly accepts it. A derived note can **never** override an
  authoritative fact (Guardrail G1).

```
authoritative doc â†’ Haiku extracts â†’ commit PROVISIONAL (low confidence)
                                       â†“  PM: [âś“ Confirm] [âśŽ Edit] [âś— Reject]
                                    confirmed â†’ full-trust fact

derived note      â†’ Haiku extracts â†’ PENDING (not in fact store)
                                       â†“  PM: [âś“ Accept] [âśŽ Edit] [âś— Reject]
                                    accepted â†’ derived-tier fact
```

| Tier | Source | On extraction | Used in validation before confirm? |
|---|---|---|---|
| `authoritative` | SOW, QA reports, ADRs | Auto-commit as `provisional` | Yes â€” but reduced confidence + unconfirmed flag |
| `derived` | AI meeting notes | Held in pending | No â€” not used until accepted |

Numeric metrics are the most error-prone (the STT "6 to 8 â†’ 628" lesson), so a
`provisional` numeric fact **never reaches High confidence** until confirmed,
regardless of tier.

### 1f. Provenance tiers

Every chunk tagged `tier: authoritative` or `tier: derived`. **Assignment rule:**
files under `docs/notes/` â†’ `derived`; everything else under `docs/` â†’
`authoritative` (override per-file via a manifest entry if needed).
Drives validator precedence: authoritative sources outrank derived notes.
Promotion from derived to authoritative is human-confirmed only.
