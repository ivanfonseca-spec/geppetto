## 1. Document ingest layer  (doc_loaders + kb_sync)  — NEW

Turns `./docs/` into the source of the KB, synced automatically.

### 1a. File type support — confirmed priority

| Order | Type | Library | Status |
|---|---|---|---|
| 1 | `.md` / `.txt` | built-in | include |
| 2 | `.pdf` | `pymupdf` | include |
| 3 | `.docx` | `python-docx` | include |
| 4 | `.xlsx` | `openpyxl` | optional, flag-gated, deferred |

### 1b. `doc_loaders.py`

Loader registry: extension → function returning `{text, metadata}`.
Metadata includes `{source, file_type, modified_date}`.

### 1c. `kb_sync.py`

Incremental sync driven by `.geppetto_manifest.json` in `./docs/`.
Manifest entry per file: `{sha256, mtime, size, chunk_ids[], ingested_at, tier}`.

On `sync()`:
1. Enumerate `./docs/` for supported extensions.
2. Classify: NEW / MODIFIED (hash changed) / SKIP (mtime+size fast-path) / DELETED.
3. NEW/MODIFIED: delete old chunks, re-chunk (~800 chars + overlap), re-embed,
   update manifest. DELETED: remove chunks + manifest entry.
4. Run **metric extraction** (§1d) on authoritative files — results go to
   **pending queue**, not directly to fact store.
5. Return `{new, modified, deleted, skipped, pending_extractions}` counts.

Atomic manifest writes (temp file + rename) — guards against crash mid-sync.
Use Chroma's local embeddings — no per-file OpenAI cost, works offline.

Wire-in: runs **blocking at server startup** (one-line summary printed) before
the app accepts requests. Also exposed as **`POST /api/sync`** for manual
re-trigger from the dashboard.

### 1d. Auto-discovered metric catalog

The metric catalog is **not hardcoded**. When a document is synced, a Haiku pass
over the text extracts measurable facts:

```
doc text → Haiku → [{metric_key, value, unit, as_of, source}, ...]
```

Metric keys are generated by the LLM using a consistent `domain.attribute`
naming convention (e.g., `qa.tests_passed_pct`, `release.date`).

**Key stability (load-bearing).** Auto-discovery only works if the *same concept*
gets the *same key* across documents and runs. If keys drift
(`qa.tests_passed_pct` one day, `qa.test_pass_rate` the next) the timeline
fragments into half-series and OUTDATED detection silently breaks within each.
To hold keys stable:
- Inject the **current list of known keys** into every extraction prompt — "reuse
  an existing key if the concept matches; mint a new one only if none fits."
- Normalize/alias returned keys (lowercase, canonical synonyms) before storing.
- Let the PM **merge / rename keys** in the pending tab; persist aliases so future
  extractions of a renamed concept resolve to the canonical key.

This replaces the hardcoded `METRIC_CATALOG` in `facts.py` with an open registry
that grows with the project — and naturally handles new risks, new tasks, and
new project areas as they appear in documents.

### 1e. Human-in-the-loop: provisional facts + pending queue

Auto-extracted metrics are never silently granted full trust, but handling differs
by tier to avoid a **false-alarm window** (Guardrail G3). Holding *everything*
pending would mean a correct new claim gets flagged CONTRADICTED while the update
waits for review; auto-accepting *everything* would let a Haiku misread ("6 to 8"
→ "628") become truth. The split below avoids both:

- **Authoritative docs → committed as `provisional`.** The value goes live
  immediately (no false-alarm window) but is marked `provisional` and used in
  validation at **reduced confidence with an "unconfirmed" note** in the alert. The
  PM **Confirms** (→ full trust), **Edits**, or **Rejects** it in the Pending tab.
- **Derived (AI meeting notes) → held in pending.** Nothing enters the fact store
  until the PM explicitly accepts it. A derived note can **never** override an
  authoritative fact (Guardrail G1).

```
authoritative doc → Haiku extracts → commit PROVISIONAL (low confidence)
                                       ↓  PM: [✓ Confirm] [✎ Edit] [✗ Reject]
                                    confirmed → full-trust fact

derived note      → Haiku extracts → PENDING (not in fact store)
                                       ↓  PM: [✓ Accept] [✎ Edit] [✗ Reject]
                                    accepted → derived-tier fact
```

| Tier | Source | On extraction | Used in validation before confirm? |
|---|---|---|---|
| `authoritative` | SOW, QA reports, ADRs | Auto-commit as `provisional` | Yes — but reduced confidence + unconfirmed flag |
| `derived` | AI meeting notes | Held in pending | No — not used until accepted |

Numeric metrics are the most error-prone (the STT "6 to 8 → 628" lesson), so a
`provisional` numeric fact **never reaches High confidence** until confirmed,
regardless of tier.

### 1f. Provenance tiers

Every chunk tagged `tier: authoritative` or `tier: derived`. **Assignment rule:**
files under `docs/notes/` → `derived`; everything else under `docs/` →
`authoritative` (override per-file via a manifest entry if needed).
Drives validator precedence: authoritative sources outrank derived notes.
Promotion from derived to authoritative is human-confirmed only.

## 5. Meeting-notes loop  (meeting_notes.py)  — NEW

Closes the loop so each meeting becomes context for the next — safely.

Called at `session.finalize()` after transcript + alerts are available.

**Implementation note:** The `brief` skill lives in Claude's conversational
interface and cannot be invoked from Python. The structured output is replicated
directly via a Haiku call with an explicit prompt template. The format can be
updated later without code changes.

**Two Haiku calls:**
- Call #1: generate a 2–4 word kebab-case slug from the meeting content.
- Call #2: structured brief in this format:

```markdown
## Current Status
## Key Achievements
## Upcoming Priorities
## Risks & Issues
## Decisions / Support Needed
## Action Items  (owner · due date)
```

**Filename:** `YYYY-MM-DD-<slug>.md`

**Saved to:**
- `docs/notes/` — tagged `tier: derived`; available as context for future meetings
- `meetings/` — permanent record alongside transcript and alerts

Notes written as `derived` are context only, never authoritative. Promotion to
authoritative is human-confirmed only (ties to §6 write-back).

## Design guardrails

**G1 — Provenance tiers prevent error amplification.**
Auto-ingesting AI meeting notes into the same KB the tool validates against risks
a self-reinforcing loop: one mis-summarized number ("QA 90%" when someone misspoke)
would become "truth" and flag future correct statements as CONTRADICTED.
Fix: tag content `authoritative` vs `derived`; the validator prefers authoritative
and a derived note can never override it. Promotion to authoritative is
human-confirmed only.

**G2 — Sync must not overwrite fact history.**
`kb_sync` deleting old chunks on MODIFIED is correct for prose, but would erase
the version history OUTDATED detection depends on.
Fix: prose chunks are replaced (latest snapshot); the fact store **appends dated
versions**. Prose = context; fact timeline = truth.

**G3 — Auto-extraction is provisional until a human confirms.**
Haiku may misread numbers or coin inconsistent keys, so no extraction is granted
full trust automatically. Authoritative extractions go live as `provisional` (used
at reduced confidence, flagged unconfirmed) and become full-trust only when the PM
confirms; derived extractions are held in pending and unused until accepted. This
avoids both failure modes at once: bad data never becomes silent truth, and there
is no window where a correct new claim is wrongly flagged because the update is
stuck awaiting review.

## The loop this enables

```
docs/ → sync at startup → KB current (authoritative)
meeting runs → claims validated against KB (authoritative > derived)
meeting ends → notes saved to docs/notes/ (derived) + meetings/
next startup → sync picks up notes → available as context, not as truth
PM reviews pending extractions → approved metrics enter fact store
```

Every meeting becomes context for the next — without letting AI summaries
masquerade as ground truth.
